# Introdução a Banco de Dados: SQL vs NoSQL

## O que é um banco de dados?

Um **banco de dados** é uma estrutura utilizada para armazenar, organizar, consultar e modificar informações de uma aplicação.

Exemplos de dados armazenados:

- Usuários.
    
- Produtos.
    
- Pedidos.
    
- Pagamentos.
    
- Mensagens.
    
- Eventos.
    
- Logs.
    
- Arquivos e metadados.
    

Em uma arquitetura tradicional:

```text
Client → Server → Banco de Dados
```

O client envia uma requisição ao servidor. O servidor aplica as regras de negócio e acessa o banco de dados para consultar ou alterar informações.

O client normalmente não deve acessar o banco diretamente.

---

## Por que utilizar um banco de dados?

Um banco de dados oferece recursos como:

- Persistência das informações.
    
- Consulta eficiente.
    
- Organização dos dados.
    
- Controle de concorrência.
    
- Segurança.
    
- Integridade.
    
- Recuperação após falhas.
    
- Relacionamento entre informações.
    
- Escalabilidade.
    

Sem um banco de dados, as informações poderiam ser perdidas sempre que o servidor fosse reiniciado.

---

## Banco de dados relacional

Um banco de dados relacional organiza os dados em **tabelas**.

Cada tabela é composta por:

- Colunas.
    
- Linhas.
    
- Tipos de dados.
    
- Chaves.
    
- Restrições.
    

Exemplo de tabela de usuários:

|Id|Nome|Email|
|---|---|---|
|1|Ana|[ana@email.com](mailto:ana@email.com)|
|2|João|[joao@email.com](mailto:joao@email.com)|

Cada linha representa um registro.

Cada coluna representa uma propriedade do registro.

---

## SQL

**SQL**, Structured Query Language, é a linguagem utilizada para consultar e manipular bancos de dados relacionais.

Exemplos de bancos SQL:

- PostgreSQL.
    
- MySQL.
    
- SQL Server.
    
- Oracle Database.
    
- SQLite.
    
- MariaDB.
    

Exemplo de consulta:

```sql
SELECT Id, Nome, Email
FROM Usuario
WHERE Id = 1;
```

Exemplo de inserção:

```sql
INSERT INTO Usuario (Nome, Email)
VALUES ('Ana', 'ana@email.com');
```

---

## Estrutura de tabelas

Considere as tabelas:

### Usuario

|Id|Nome|Email|
|---|---|---|
|1|Ana|[ana@email.com](mailto:ana@email.com)|
|2|João|[joao@email.com](mailto:joao@email.com)|

### Pedido

|Id|IdUsuario|Valor|
|---|---|---|
|100|1|150.00|
|101|1|80.00|
|102|2|200.00|

O campo `IdUsuario` da tabela `Pedido` aponta para um usuário.

Essa relação pode ser representada como:

```text
Usuario 1 ─── N Pedido
```

Um usuário pode possuir vários pedidos.

---

## Chave primária

A **Primary Key**, ou chave primária, identifica cada registro de forma única.

Exemplo:

```sql
CREATE TABLE Usuario
(
    Id INT PRIMARY KEY,
    Nome VARCHAR(100),
    Email VARCHAR(200)
);
```

Na tabela `Usuario`, o campo `Id` identifica cada usuário.

Não podem existir dois usuários com o mesmo valor de chave primária.

---

## Chave estrangeira

A **Foreign Key**, ou chave estrangeira, representa o relacionamento entre tabelas.

Exemplo:

```sql
CREATE TABLE Pedido
(
    Id INT PRIMARY KEY,
    IdUsuario INT NOT NULL,
    Valor DECIMAL(10, 2) NOT NULL,
    FOREIGN KEY (IdUsuario) REFERENCES Usuario(Id)
);
```

A chave estrangeira garante que um pedido não seja relacionado a um usuário inexistente.

---

## Relacionamentos

Os principais relacionamentos em bancos relacionais são:

### Um para um

```text
Pessoa 1 ─── 1 Passaporte
```

Uma pessoa possui um passaporte e um passaporte pertence a uma pessoa.

### Um para muitos

```text
Usuario 1 ─── N Pedido
```

Um usuário pode possuir vários pedidos.

### Muitos para muitos

```text
Aluno N ─── N Curso
```

Um aluno pode participar de vários cursos e um curso pode possuir vários alunos.

Esse relacionamento normalmente utiliza uma tabela intermediária.

```text
Aluno
AlunoCurso
Curso
```

---

## Schema

O **schema** define a estrutura dos dados.

Ele pode determinar:

- Nome das tabelas.
    
- Nome das colunas.
    
- Tipo de cada coluna.
    
- Campos obrigatórios.
    
- Relacionamentos.
    
- Restrições.
    
- Valores únicos.
    

Exemplo:

```sql
CREATE TABLE Produto
(
    Id INT PRIMARY KEY,
    Nome VARCHAR(150) NOT NULL,
    Preco DECIMAL(10, 2) NOT NULL,
    Estoque INT NOT NULL
);
```

Em bancos SQL, o schema normalmente é definido antes da inserção dos dados.

Isso é chamado de **schema on write**.

O dado precisa respeitar a estrutura no momento em que é gravado.

---

## Normalização

A normalização é uma técnica utilizada para reduzir duplicação e inconsistência de dados.

Exemplo de estrutura não normalizada:

|IdPedido|Cliente|EmailCliente|Produto|Valor|
|---|---|---|---|---|
|1|Ana|[ana@email.com](mailto:ana@email.com)|Notebook|3500|
|2|Ana|[ana@email.com](mailto:ana@email.com)|Mouse|100|

As informações da cliente Ana são repetidas.

Uma estrutura normalizada poderia separar os dados:

```text
Usuario
Pedido
Produto
PedidoProduto
```

Vantagens:

- Reduz duplicação.
    
- Evita inconsistências.
    
- Facilita atualizações.
    
- Melhora a integridade.
    

Desvantagens:

- Pode exigir mais joins.
    
- Consultas podem ficar mais complexas.
    
- Pode aumentar o custo de leitura.
    

---

## JOIN

O `JOIN` permite combinar dados de diferentes tabelas.

Exemplo:

```sql
SELECT
    Usuario.Nome,
    Pedido.Id,
    Pedido.Valor
FROM Pedido
INNER JOIN Usuario
    ON Usuario.Id = Pedido.IdUsuario;
```

Resultado:

|Nome|Id|Valor|
|---|---|---|
|Ana|100|150.00|
|Ana|101|80.00|
|João|102|200.00|

Os joins são uma característica importante dos bancos relacionais.

---

## Transação

Uma transação agrupa várias operações que devem ser executadas como uma única unidade.

Exemplo de transferência bancária:

```text
1. Remover R$ 100 da conta A.
2. Adicionar R$ 100 à conta B.
```

As duas operações precisam acontecer juntas.

Caso apenas a primeira aconteça, o dinheiro desapareceria do sistema.

Exemplo:

```sql
BEGIN TRANSACTION;

UPDATE Conta
SET Saldo = Saldo - 100
WHERE Id = 1;

UPDATE Conta
SET Saldo = Saldo + 100
WHERE Id = 2;

COMMIT;
```

Caso ocorra algum problema:

```sql
ROLLBACK;
```

O rollback desfaz as alterações da transação.

---

## ACID

Bancos relacionais são frequentemente associados às propriedades **ACID**.

### Atomicity

A transação acontece por completo ou não acontece.

```text
Todas as operações são executadas
ou
nenhuma operação é executada
```

### Consistency

A transação leva o banco de um estado válido para outro estado válido.

As regras e restrições precisam continuar sendo respeitadas.

### Isolation

Transações concorrentes não devem interferir incorretamente umas nas outras.

Cada transação deve se comportar como se estivesse sendo executada de forma isolada, conforme o nível de isolamento utilizado.

### Durability

Depois que uma transação é confirmada, os dados devem permanecer armazenados, mesmo se ocorrer uma falha.

---

## Vantagens dos bancos SQL

- Estrutura bem definida.
    
- Relacionamentos entre dados.
    
- Suporte a joins.
    
- Transações robustas.
    
- Integridade referencial.
    
- Linguagem de consulta padronizada.
    
- Bons para dados altamente relacionados.
    
- Maturidade e grande ecossistema.
    
- Adequados para operações financeiras.
    

---

## Desvantagens dos bancos SQL

- Mudanças de schema podem exigir migrations.
    
- Escalabilidade horizontal pode ser mais complexa.
    
- Muitos joins podem aumentar o custo das consultas.
    
- Estruturas muito flexíveis podem ser difíceis de modelar.
    
- Normalização excessiva pode prejudicar leituras complexas.
    

Essas limitações dependem do banco, da modelagem e da arquitetura utilizada.

---

# NoSQL

## O que é NoSQL?

**NoSQL** significa, de forma geral, **Not Only SQL**.

NoSQL representa um grupo de bancos que não seguem obrigatoriamente o modelo relacional tradicional de tabelas, linhas e relacionamentos.

Eles podem armazenar dados em diferentes formatos, como:

- Documentos.
    
- Chave-valor.
    
- Colunas.
    
- Grafos.
    

NoSQL não representa uma única tecnologia ou um único modelo de banco.

---

## Por que bancos NoSQL foram criados?

Bancos NoSQL ganharam popularidade por necessidades como:

- Grandes volumes de dados.
    
- Alta quantidade de requisições.
    
- Escalabilidade horizontal.
    
- Estruturas de dados flexíveis.
    
- Distribuição geográfica.
    
- Baixa latência.
    
- Dados com formatos variados.
    

Eles são frequentemente utilizados em sistemas distribuídos de grande escala.

---

## Schema flexível

Muitos bancos NoSQL permitem que documentos da mesma coleção possuam estruturas diferentes.

Exemplo:

```json
{
  "id": 1,
  "nome": "Ana",
  "email": "ana@email.com"
}
```

Outro documento da mesma coleção:

```json
{
  "id": 2,
  "nome": "João",
  "telefone": "81999999999",
  "enderecos": [
    {
      "cidade": "Recife",
      "estado": "PE"
    }
  ]
}
```

Essa flexibilidade pode facilitar a evolução da aplicação.

Entretanto, schema flexível não significa ausência de estrutura.

A aplicação ainda precisa saber:

- Quais campos existem.
    
- Quais campos são obrigatórios.
    
- Quais tipos são esperados.
    
- Como validar os documentos.
    
- Como tratar documentos antigos.
    

Em alguns bancos NoSQL, também é possível aplicar validação de schema.

---

# Tipos de bancos NoSQL

## Banco de documentos

Armazena os dados como documentos, normalmente semelhantes a JSON.

Exemplos:

- MongoDB.
    
- Couchbase.
    
- CouchDB.
    
- Amazon DocumentDB.
    

Exemplo:

```json
{
  "id": 100,
  "cliente": {
    "id": 1,
    "nome": "Ana"
  },
  "itens": [
    {
      "produto": "Notebook",
      "quantidade": 1,
      "valor": 3500
    },
    {
      "produto": "Mouse",
      "quantidade": 2,
      "valor": 100
    }
  ],
  "valorTotal": 3700
}
```

O documento pode armazenar informações relacionadas dentro da mesma estrutura.

### Casos de uso

- Catálogos de produtos.
    
- Perfis de usuários.
    
- Sistemas de conteúdo.
    
- Aplicações com estrutura variável.
    
- Carrinhos de compra.
    
- Dados semelhantes a JSON.
    

### Vantagens

- Estrutura flexível.
    
- Representação próxima dos objetos da aplicação.
    
- Reduz a necessidade de joins.
    
- Fácil armazenamento de dados aninhados.
    

### Desvantagens

- Pode gerar duplicação.
    
- Atualizar dados repetidos pode ser difícil.
    
- Relacionamentos complexos podem ser problemáticos.
    
- Documentos muito grandes podem prejudicar o desempenho.
    

---

## Banco chave-valor

Armazena informações como um par composto por uma chave e um valor.

```text
Chave → Valor
```

Exemplo:

```text
session:user:123 → dados da sessão
```

O valor pode ser:

- Texto.
    
- Número.
    
- JSON.
    
- Lista.
    
- Estrutura binária.
    

Exemplos:

- Redis.
    
- Amazon DynamoDB, em parte de seu modelo.
    
- Riak.
    

### Casos de uso

- Cache.
    
- Sessões.
    
- Carrinho de compras.
    
- Contadores.
    
- Rate limiting.
    
- Dados temporários.
    
- Filas simples.
    

### Vantagens

- Consultas muito rápidas pela chave.
    
- Modelo simples.
    
- Boa escalabilidade.
    
- Baixa latência.
    

### Desvantagens

- Consultas complexas são limitadas.
    
- Busca por campos internos pode não ser eficiente.
    
- Relacionamentos não são o foco.
    

---

## Banco orientado a colunas

Também chamado de **wide-column database**.

Os dados são organizados em famílias de colunas, permitindo grande flexibilidade e distribuição.

Exemplos:

- Apache Cassandra.
    
- Apache HBase.
    
- Google Cloud Bigtable.
    
- ScyllaDB.
    

Exemplo conceitual:

```text
Usuario:1
    nome: Ana
    email: ana@email.com
    ultimo_login: 2026-07-11
```

### Casos de uso

- Grandes volumes de dados.
    
- Eventos.
    
- Logs.
    
- Métricas.
    
- Dados de sensores.
    
- Escritas em grande escala.
    
- Sistemas distribuídos globalmente.
    

### Vantagens

- Alta capacidade de escrita.
    
- Escalabilidade horizontal.
    
- Boa distribuição de dados.
    
- Adequado para grandes volumes.
    

### Desvantagens

- Modelagem orientada às consultas.
    
- Consultas ad hoc podem ser difíceis.
    
- Relacionamentos complexos não são adequados.
    
- Pode exigir duplicação de dados.
    

---

## Banco de grafos

Armazena dados como:

- Nós.
    
- Arestas.
    
- Propriedades.
    

Exemplo:

```text
Pessoa A ── segue ──> Pessoa B
Pessoa B ── trabalha em ──> Empresa X
```

Exemplos:

- Neo4j.
    
- Amazon Neptune.
    
- ArangoDB.
    
- JanusGraph.
    

### Casos de uso

- Redes sociais.
    
- Sistemas de recomendação.
    
- Detecção de fraude.
    
- Rotas.
    
- Relações organizacionais.
    
- Redes de conhecimento.
    

### Vantagens

- Excelente para relacionamentos complexos.
    
- Travessia eficiente entre conexões.
    
- Representação natural de redes.
    

### Desvantagens

- Não é ideal para qualquer tipo de dado.
    
- Requer uma modelagem diferente.
    
- Pode possuir menor familiaridade e ecossistema em algumas equipes.
    

---

# SQL vs NoSQL

## Comparação geral

|Característica|SQL|NoSQL|
|---|---|---|
|Modelo|Tabelas e relacionamentos|Documentos, chave-valor, colunas ou grafos|
|Schema|Geralmente rígido|Geralmente flexível|
|Relacionamentos|Muito forte|Depende do tipo de banco|
|JOIN|Recurso comum|Normalmente limitado ou evitado|
|Transações|Forte suporte|Varia conforme o banco|
|Escalabilidade|Tradicionalmente vertical|Frequentemente horizontal|
|Consistência|Geralmente forte|Pode priorizar disponibilidade|
|Consultas complexas|Muito adequado|Depende do modelo|
|Dados aninhados|Menos natural|Muito comum em documentos|
|Duplicação de dados|Normalmente reduzida|Frequentemente aceita|
|Modelagem|Orientada às entidades e relações|Frequentemente orientada às consultas|

Essa comparação é uma generalização.

Bancos SQL modernos também podem escalar horizontalmente, trabalhar com JSON e operar de forma distribuída.

Bancos NoSQL modernos também podem suportar transações, schemas e consistência forte.

---

## Escalabilidade vertical e horizontal

### Escalabilidade vertical

Aumenta os recursos da máquina existente.

```text
Mais CPU
Mais memória
Disco mais rápido
```

Bancos SQL tradicionalmente utilizam bastante escalabilidade vertical.

### Escalabilidade horizontal

Adiciona novas máquinas ao sistema.

```text
Database 1
Database 2
Database 3
```

Bancos NoSQL normalmente são projetados com escalabilidade horizontal como uma característica central.

Contudo, tanto SQL quanto NoSQL podem utilizar os dois tipos de escalabilidade.

---

## Sharding

O **sharding** divide os dados entre diferentes servidores.

Exemplo:

```text
Shard 1 → Usuários de 1 até 1.000.000
Shard 2 → Usuários de 1.000.001 até 2.000.000
Shard 3 → Usuários de 2.000.001 até 3.000.000
```

Outra possibilidade é dividir por região:

```text
Shard Brasil
Shard Estados Unidos
Shard Europa
```

A chave utilizada para decidir onde o dado será armazenado é chamada de **shard key**.

Uma shard key ruim pode causar distribuição desigual.

Exemplo:

```text
Shard 1 → 90% das requisições
Shard 2 → 5% das requisições
Shard 3 → 5% das requisições
```

Isso cria um **hotspot**.

---

## Replicação

A replicação mantém cópias dos dados em diferentes servidores.

```text
Primary
├── Replica 1
├── Replica 2
└── Replica 3
```

A replicação pode melhorar:

- Disponibilidade.
    
- Tolerância a falhas.
    
- Capacidade de leitura.
    
- Recuperação de desastres.
    

Uma arquitetura comum permite:

```text
Escritas → Primary
Leituras → Replicas
```

Entretanto, as réplicas podem demorar para receber os dados mais recentes.

Esse atraso é chamado de **replication lag**.

---

## Consistência forte

Na consistência forte, uma leitura realizada após uma escrita deve retornar o valor mais recente.

```text
Escreve A
→ Lê
→ Recebe A
```

É importante em sistemas como:

- Pagamentos.
    
- Saldos bancários.
    
- Controle de estoque crítico.
    
- Reservas.
    
- Transferências.
    

---

## Consistência eventual

Na consistência eventual, diferentes réplicas podem apresentar valores diferentes durante um intervalo.

Com o tempo, todas convergem para o mesmo estado.

```text
Escrita realizada
→ Replica 1 atualizada
→ Replica 2 ainda desatualizada
→ Após algum tempo, todas ficam iguais
```

É aceitável em situações como:

- Número de curtidas.
    
- Contagem de visualizações.
    
- Feed de atividades.
    
- Métricas.
    
- Algumas recomendações.
    

A consistência eventual não significa que o sistema nunca ficará consistente.

Significa que a propagação da atualização pode levar algum tempo.

---

## Disponibilidade

Disponibilidade significa que o sistema consegue continuar respondendo às requisições.

Em sistemas distribuídos, pode existir um conflito entre:

- Retornar uma resposta rapidamente.
    
- Garantir que todos os dados estejam totalmente atualizados.
    

Esse conflito aparece principalmente durante falhas de rede.

---

## Teorema CAP

O teorema CAP afirma que um sistema distribuído, durante uma partição de rede, precisa escolher entre:

- **Consistency:** consistência.
    
- **Availability:** disponibilidade.
    
- **Partition Tolerance:** tolerância a partições.
    

### Consistency

Todos os nós retornam o mesmo dado atualizado.

### Availability

Toda requisição recebe uma resposta, mesmo que ela não contenha o dado mais recente.

### Partition Tolerance

O sistema continua operando mesmo quando alguns nós não conseguem se comunicar.

Em sistemas distribuídos reais, partições de rede podem acontecer.

Por isso, normalmente a tolerância a partições precisa ser considerada.

Durante uma partição, o sistema tende a priorizar:

```text
CP → Consistência + tolerância a partições
```

ou:

```text
AP → Disponibilidade + tolerância a partições
```

CAP não significa que um banco é sempre CP ou sempre AP em qualquer situação.

O comportamento pode depender da operação, da configuração e do nível de consistência solicitado.

---

## Exemplo de escolha CP

Considere duas pessoas tentando comprar o último ingresso de um evento.

O sistema precisa garantir que apenas uma compra seja aprovada.

Nesse cenário, pode ser melhor recusar temporariamente uma operação do que vender o mesmo ingresso duas vezes.

```text
Prioridade: consistência
```

---

## Exemplo de escolha AP

Considere um sistema de curtidas.

Mesmo durante uma falha de comunicação entre regiões, o sistema pode continuar aceitando curtidas.

As contagens podem ficar temporariamente diferentes, mas serão sincronizadas depois.

```text
Prioridade: disponibilidade
```

---

## BASE

Alguns bancos NoSQL são associados ao modelo **BASE**.

### Basically Available

O sistema busca permanecer disponível.

### Soft State

O estado pode mudar ao longo do tempo devido à sincronização entre réplicas.

### Eventual Consistency

Os dados convergem para um estado consistente após algum tempo.

ACID e BASE não são necessariamente opostos absolutos.

Sistemas modernos podem combinar características dos dois modelos.

---

# Modelagem em SQL

Em um banco relacional, um sistema de pedidos poderia ser modelado assim:

```text
Usuario
Pedido
PedidoItem
Produto
Pagamento
```

Exemplo:

```sql
CREATE TABLE Pedido
(
    Id INT PRIMARY KEY,
    IdUsuario INT NOT NULL,
    DataCriacao DATETIME NOT NULL,
    ValorTotal DECIMAL(10, 2) NOT NULL,
    FOREIGN KEY (IdUsuario) REFERENCES Usuario(Id)
);
```

Os dados são separados para reduzir duplicação.

---

# Modelagem em banco de documentos

Em um banco de documentos, um pedido poderia ser armazenado em um único documento:

```json
{
  "id": 100,
  "usuario": {
    "id": 1,
    "nome": "Ana"
  },
  "itens": [
    {
      "idProduto": 10,
      "nome": "Notebook",
      "quantidade": 1,
      "preco": 3500
    }
  ],
  "valorTotal": 3500,
  "dataCriacao": "2026-07-11"
}
```

O nome do produto foi duplicado dentro do pedido.

Essa duplicação pode ser intencional.

Ela preserva o nome que o produto possuía no momento da compra.

---

## Normalização vs desnormalização

### Normalização

Os dados são separados para evitar repetição.

```text
Produto armazenado apenas na tabela Produto
```

Vantagens:

- Menor duplicação.
    
- Atualizações mais centralizadas.
    
- Melhor integridade.
    

Desvantagens:

- Mais joins.
    
- Consultas podem ser mais custosas.
    

### Desnormalização

Os dados são duplicados para facilitar a leitura.

```text
Nome do produto armazenado também no Pedido
```

Vantagens:

- Leituras mais rápidas.
    
- Menos joins.
    
- Dados necessários ficam próximos.
    

Desvantagens:

- Mais duplicação.
    
- Atualizações podem ser mais difíceis.
    
- Risco de inconsistência.
    

Bancos NoSQL frequentemente utilizam desnormalização.

Isso não significa que a desnormalização seja exclusiva de NoSQL.

---

## Modelagem orientada às consultas

Em bancos SQL, a modelagem normalmente começa pelas entidades e relacionamentos.

```text
Quais entidades existem?
Como elas se relacionam?
```

Em muitos bancos NoSQL, a modelagem começa pelas consultas.

```text
Quais consultas o sistema precisa executar?
Como os dados devem ser armazenados para atender essas consultas?
```

Exemplo:

Se o sistema sempre consulta todos os itens de um pedido juntos, pode ser vantajoso armazená-los no mesmo documento.

---

# Quando utilizar SQL?

SQL pode ser uma boa escolha quando:

- Os dados possuem relacionamentos complexos.
    
- É necessário realizar joins.
    
- Transações são essenciais.
    
- A consistência é uma prioridade.
    
- O schema é relativamente estável.
    
- A integridade dos dados é importante.
    
- Existem consultas variadas e complexas.
    
- O sistema envolve pagamentos ou dados financeiros.
    
- A equipe possui maior experiência com bancos relacionais.
    

Exemplos:

- Sistemas bancários.
    
- E-commerce.
    
- ERP.
    
- Sistemas de faturamento.
    
- Controle financeiro.
    
- Gestão acadêmica.
    
- Reservas.
    
- Sistemas administrativos.
    

---

# Quando utilizar NoSQL?

NoSQL pode ser uma boa escolha quando:

- Existe grande volume de dados.
    
- A aplicação exige alta escalabilidade horizontal.
    
- A estrutura dos dados muda frequentemente.
    
- Os dados não possuem muitos relacionamentos.
    
- As consultas são conhecidas e bem definidas.
    
- É necessário obter baixa latência.
    
- O sistema possui grande volume de escrita.
    
- Os dados são naturalmente documentos, grafos ou chave-valor.
    

Exemplos:

- Cache.
    
- Sessões.
    
- Feeds.
    
- Logs.
    
- Métricas.
    
- Chats.
    
- Catálogos.
    
- Sistemas de recomendação.
    
- Dados de sensores.
    
- Redes sociais.

# SQL ou NoSQL?

A escolha não deve ser baseada apenas em popularidade.

Perguntas importantes:

- Quais dados serão armazenados?
    
- Como os dados se relacionam?
    
- Quais consultas serão realizadas?
    
- Qual é o volume de leitura?
    
- Qual é o volume de escrita?
    
- É necessário realizar transações?
    
- Qual nível de consistência é necessário?
    
- O sistema precisa de escalabilidade horizontal?
    
- A estrutura dos dados muda frequentemente?
    
- A equipe domina qual tecnologia?
    
- Qual é a consequência de dados temporariamente desatualizados?
    
- Existe necessidade de consultas ad hoc?
    
- Qual é o custo operacional?
    

---

## Não existe banco perfeito

Cada banco possui vantagens, limitações e casos de uso.

A escolha deve considerar:

```text
Requisitos do sistema
+ padrão de acesso
+ volume de dados
+ consistência
+ escalabilidade
+ experiência da equipe
+ custo operacional
```

Não se deve escolher NoSQL apenas porque o sistema pode crescer.

Também não se deve escolher SQL para qualquer situação apenas por ser mais conhecido.

---

# Polyglot Persistence

**Polyglot Persistence** significa utilizar diferentes tipos de bancos no mesmo sistema.

Exemplo de um e-commerce:

```text
PostgreSQL → pedidos e pagamentos
Redis → cache e sessões
Elasticsearch → busca de produtos
MongoDB → catálogo flexível
Neo4j → recomendações
```

Cada banco é utilizado para o problema em que apresenta melhores características.

Vantagens:

- Tecnologia adequada para cada caso.
    
- Melhor desempenho.
    
- Maior flexibilidade.
    

Desvantagens:

- Mais complexidade.
    
- Mais tecnologias para manter.
    
- Maior custo operacional.
    
- Mais conhecimento necessário.
    
- Sincronização entre bancos.
    

Um sistema não deve utilizar vários bancos sem necessidade real.

---

# Exemplo prático: rede social

Uma rede social poderia utilizar:

```text
SQL:
- Usuários.
- Assinaturas.
- Pagamentos.
- Configurações de conta.

Banco de documentos:
- Publicações.
- Perfis flexíveis.

Banco de grafos:
- Seguidores.
- Relações entre usuários.

Chave-valor:
- Sessões.
- Cache.
- Contagem temporária.

Wide-column:
- Feed.
- Eventos.
- Métricas.
```

Essa divisão depende da escala e dos requisitos reais.

Em sistemas menores, um único banco relacional pode atender a maior parte das necessidades.

# Erros comuns

## Escolher NoSQL apenas para escalar

Bancos SQL também podem:

- Utilizar réplicas.
    
- Utilizar particionamento.
    
- Utilizar sharding.
    
- Escalar horizontalmente.
    
- Armazenar JSON.
    

A escolha precisa considerar todo o contexto.

## Ignorar os padrões de consulta

Um banco pode armazenar os dados corretamente, mas responder mal às consultas reais do sistema.

A modelagem precisa considerar como os dados serão lidos.

## Criar relacionamentos complexos em banco inadequado

Tentar representar uma grande quantidade de relacionamentos em um banco chave-valor pode aumentar muito a complexidade da aplicação.

## Duplicar dados sem estratégia

A desnormalização pode melhorar a leitura, mas exige uma estratégia para manter os dados sincronizados.

## Aplicar joins em grande escala sem avaliar o custo

Joins podem ser muito úteis, mas podem ficar caros com grandes volumes, falta de índices ou modelagem inadequada.

## Adotar muitas tecnologias cedo demais

Um sistema pequeno pode começar com um banco relacional e evoluir conforme os gargalos aparecem.

# Índices

Um índice é uma estrutura utilizada para acelerar consultas.

Exemplo:

```sql
CREATE INDEX IX_Usuario_Email
ON Usuario(Email);
```

Sem índice, o banco pode precisar percorrer todos os registros.

```text
Table Scan
```

Com índice, ele pode localizar o dado de forma mais eficiente.

```text
Index Seek
```

Índices melhoram leituras, mas possuem custos:

- Ocupam armazenamento.
    
- Aumentam o custo das inserções.
    
- Aumentam o custo das atualizações.
    
- Aumentam o custo das exclusões.
    

Esse conceito é importante tanto em bancos SQL quanto NoSQL.

# Resumo SQL

```text
SQL:
- Tabelas.
- Linhas e colunas.
- Schema bem definido.
- Relacionamentos.
- Joins.
- Transações.
- Integridade referencial.
- Normalização.
- Consultas complexas.
```

É indicado principalmente para dados estruturados, relacionados e operações que exigem forte consistência.

---

# Resumo NoSQL

```text
NoSQL:
- Documentos.
- Chave-valor.
- Wide-column.
- Grafos.
- Schema geralmente flexível.
- Desnormalização.
- Modelagem orientada às consultas.
- Escalabilidade horizontal.
- Alta distribuição.
```

É indicado principalmente para grandes volumes, estruturas flexíveis, alta escala e padrões de acesso específicos.

---

# Comparação resumida

```text
SQL:
- Prioriza relações, integridade e consultas complexas.
- Geralmente utiliza normalização.
- Possui forte suporte a transações.

NoSQL:
- Prioriza flexibilidade, distribuição e padrões específicos de acesso.
- Geralmente aceita desnormalização.
- O comportamento de consistência e transações varia entre bancos.
```

---

# Pontos importantes para System Design

Durante uma entrevista ou projeto de System Design, não basta dizer apenas:

```text
Vou usar SQL.
```

ou:

```text
Vou usar NoSQL porque escala mais.
```

É necessário justificar a decisão.

Exemplo de justificativa para SQL:

```text
Vou utilizar PostgreSQL porque pedidos, pagamentos e usuários possuem relações importantes. O sistema exige transações e consistência forte para evitar cobranças ou reservas duplicadas.
```

Exemplo de justificativa para NoSQL:

```text
Vou utilizar um banco de documentos para o catálogo porque os produtos possuem atributos muito diferentes entre categorias e são lidos como documentos completos.
```

Exemplo de justificativa para Redis:

```text
Vou utilizar Redis para armazenar sessões e dados temporários, pois o acesso é feito diretamente por chave e exige baixa latência.
```

```text
1. Os dados possuem muitos relacionamentos?
2. Preciso de transações?
3. Preciso de consistência forte?
4. Qual é o volume de leitura?
5. Qual é o volume de escrita?
6. Quais consultas serão realizadas?
7. O schema muda frequentemente?
8. Posso aceitar duplicação?
9. Posso aceitar consistência eventual?
10. Preciso escalar horizontalmente?
11. Existe necessidade de distribuição geográfica?

```

A escolha do banco deve ser consequência dos requisitos, e não o ponto de partida do design.
