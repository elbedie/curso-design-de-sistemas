# Módulo 12 — Requisitos e Estimativas

Antes de escolher bancos, caches, filas ou qualquer tecnologia, é necessário entender o problema que está sendo resolvido.

Uma arquitetura só pode ser avaliada quando existem requisitos claros.

Não existe sistema simplesmente “escalável”, “rápido” ou “altamente disponível”.

Essas características precisam ser transformadas em metas mensuráveis.

Exemplos:

```text
Disponibilidade:
99,95% ao mês

Latência:
P95 abaixo de 200 ms

Escala:
50 mil requisições por segundo

Durabilidade:
perda máxima de 1 minuto de dados
```

Este módulo apresenta:

* Como levantar requisitos funcionais.
* Como definir requisitos não funcionais.
* Como trabalhar com disponibilidade.
* Como estimar escalabilidade.
* Como definir metas de latência.
* Como calcular throughput.
* Como estimar armazenamento.
* Como estimar banda.
* Como estimar memória e cache.
* Como realizar back-of-the-envelope calculations.
* Como comunicar hipóteses durante entrevistas de System Design.

# 1. Por que requisitos vêm antes da arquitetura

Uma decisão arquitetural só faz sentido dentro de um contexto.

Considere dois sistemas.

## Sistema A

```text
1.000 usuários
10 requisições por segundo
Uso interno
Indisponibilidade de alguns minutos é aceitável
```

## Sistema B

```text
100 milhões de usuários
500 mil requisições por segundo
Operação global
Indisponibilidade causa perdas financeiras
```

Os dois sistemas não precisam da mesma arquitetura.

O Sistema A pode funcionar bem com:

```text
Aplicação
   |
   v
SQL Server
```

O Sistema B provavelmente precisará de:

```text
CDN
Load Balancers
Múltiplas regiões
Cache
Filas
Particionamento
Replicação
Observabilidade
```

Adicionar a arquitetura do Sistema B ao Sistema A pode gerar:

* Custo desnecessário.
* Maior complexidade.
* Mais pontos de falha.
* Operação mais difícil.
* Menor velocidade de desenvolvimento.

> A arquitetura deve ser proporcional aos requisitos.

# 2. Requisitos funcionais

Requisitos funcionais descrevem o que o sistema deve fazer.

Exemplo para uma rede social:

* Usuário pode criar uma conta.
* Usuário pode publicar conteúdo.
* Usuário pode seguir outras pessoas.
* Usuário pode visualizar um feed.
* Usuário pode curtir publicações.
* Usuário pode comentar.
* Usuário pode excluir uma publicação.

Exemplo para um sistema de pagamentos:

* Criar pagamento.
* Autorizar pagamento.
* Capturar pagamento.
* Cancelar pagamento.
* Reembolsar pagamento.
* Consultar status.
* Enviar notificação.

## Priorização

Em entrevistas, não é necessário projetar todas as funcionalidades.

Defina o escopo principal.

Exemplo:

```text
Escopo principal:
- Criar publicação.
- Seguir usuário.
- Gerar feed.

Fora do escopo:
- Mensagens privadas.
- Vídeos ao vivo.
- Publicidade.
```

Isso evita gastar tempo em funcionalidades periféricas.

# 3. Requisitos não funcionais

Requisitos não funcionais descrevem como o sistema deve se comportar.

Principais categorias:

* Disponibilidade.
* Escalabilidade.
* Latência.
* Throughput.
* Durabilidade.
* Consistência.
* Segurança.
* Observabilidade.
* Manutenibilidade.
* Recuperação de desastre.
* Custo.

Exemplo:

```text
Disponibilidade:
99,99%

Latência:
P95 abaixo de 200 ms

Escala:
100 mil leituras por segundo

Durabilidade:
nenhum pagamento confirmado pode ser perdido

Consistência:
forte para saldo
eventual para feed
```

# 4. Requisitos mensuráveis

Requisitos vagos são difíceis de validar.

## Requisito vago

```text
O sistema deve ser rápido.
```

## Requisito mensurável

```text
95% das requisições devem responder em até 200 ms.
99% devem responder em até 500 ms.
```

## Requisito vago

```text
O sistema deve ser altamente disponível.
```

## Requisito mensurável

```text
O serviço deve possuir disponibilidade mensal de 99,95%.
```

## Requisito vago

```text
O sistema deve escalar.
```

## Requisito mensurável

```text
O sistema deve suportar 50 mil requisições por segundo
e picos de até 120 mil requisições por segundo.
```

## Regra prática

Sempre transforme adjetivos em números.

```text
Rápido           --> latência
Escalável        --> throughput e volume
Disponível       --> porcentagem de uptime
Durável          --> probabilidade de perda
Recuperável      --> RTO e RPO
```

# 5. Disponibilidade

Disponibilidade representa a proporção de tempo em que o sistema está operacional.

Fórmula:

```text
Disponibilidade =
Tempo operacional / Tempo total
```

Exemplo:

```text
Tempo total:
30 dias

Indisponibilidade:
21,6 minutos
```

A disponibilidade aproximada é:

```text
99,95%
```

## Disponível para quem?

Disponibilidade deve ser definida do ponto de vista do usuário.

Um processo pode estar vivo, mas o sistema indisponível.

Exemplo:

```text
API está executando
SQL Server está indisponível
Usuário não consegue concluir pedido
```

Para o usuário:

```text
Sistema indisponível
```

---

# 6. Cálculo de disponibilidade

Considere um mês de 30 dias.

```text
30 dias
x 24 horas
x 60 minutos
=
43.200 minutos
```

Com 99,9% de disponibilidade:

```text
Indisponibilidade permitida =
43.200 x 0,001
=
43,2 minutos
```

Com 99,99%:

```text
43.200 x 0,0001
=
4,32 minutos
```

## Fórmula

```text
Downtime permitido =
Período total x (1 - disponibilidade)
```

---

# 7. Tabela de disponibilidade

Valores aproximados:

| Disponibilidade |       Downtime anual |      Downtime mensal |
| --------------: | -------------------: | -------------------: |
|             99% |    3 dias e 15 horas | 7 horas e 12 minutos |
|           99,9% | 8 horas e 46 minutos |           43 minutos |
|          99,95% | 4 horas e 23 minutos |           22 minutos |
|          99,99% |           52 minutos |            4 minutos |
|         99,999% |            5 minutos |          26 segundos |

## Trade-off

Cada “nove” adicional pode aumentar significativamente:

* Custo.
* Complexidade.
* Redundância.
* Operação.
* Testes.
* Número de regiões.
* Automação.
* Capacidade ociosa.

Não escolha 99,999% sem necessidade real.

# 8. Disponibilidade composta

Um sistema depende de múltiplos componentes.

Exemplo:

```text
Cliente
   |
   v
API Gateway
   |
   v
Orders API
   |
   v
SQL Server
```

Se todos forem necessários para a operação, a disponibilidade total é aproximadamente o produto das disponibilidades.

```text
Disponibilidade total =
Gateway
x API
x Banco
```

Exemplo:

```text
Gateway: 99,99%
API:     99,99%
Banco:   99,99%
```

```text
0,9999 x 0,9999 x 0,9999
=
0,99970003
```

Disponibilidade total aproximada:

```text
99,97%
```

Mesmo que cada componente possua 99,99%, o sistema completo pode ter menos.

# 9. Dependências em série

Dependências em série são aquelas em que todas precisam estar disponíveis.

```text
A --> B --> C
```

Disponibilidade:

```text
A x B x C
```

Exemplo:

```text
Authentication Service
Orders Service
Payments Service
```

Se uma requisição de checkout precisar dos três de forma síncrona, a disponibilidade é limitada pelo conjunto.

## Consequência

Adicionar dependências síncronas pode reduzir:

* Disponibilidade.
* Latência.
* Resiliência.

## Alternativas

* Cache.
* Processamento assíncrono.
* Fallback.
* Dados locais.
* Graceful degradation.
* Remover chamadas do caminho crítico.

---

# 10. Redundância e dependências em paralelo

Considere dois servidores idênticos.

```text
Load Balancer
   |
   +--> Server A
   +--> Server B
```

Se o sistema continua funcionando enquanto pelo menos um está disponível, a disponibilidade melhora.

Probabilidade de ambos falharem:

```text
Falha de A x Falha de B
```

Se cada servidor tem 99% de disponibilidade:

```text
Falha individual:
1%

Falha conjunta:
1% x 1%
=
0,01%
```

Disponibilidade conjunta:

```text
99,99%
```

## Cuidado

Essa conta pressupõe falhas independentes.

Na prática, os dois servidores podem compartilhar:

* Mesma região.
* Mesmo banco.
* Mesmo deploy.
* Mesma configuração.
* Mesmo provedor.
* Mesmo erro de software.

Falhas correlacionadas reduzem o benefício da redundância.

# 11. SLA, SLO e SLI

## SLI

Service Level Indicator.

É uma métrica observada.

Exemplos:

* Disponibilidade medida.
* Latência P95.
* Taxa de sucesso.
* Taxa de mensagens processadas.

```text
SLI:
99,96% das requisições foram bem-sucedidas.
```

## SLO

Service Level Objective.

É uma meta interna.

```text
SLO:
99,95% de sucesso mensal.
```

## SLA

Service Level Agreement.

É um compromisso contratual com consequências.

```text
SLA:
99,9% de disponibilidade.
Caso contrário, haverá crédito ao cliente.
```

## Relação

```text
SLI:
o que foi medido

SLO:
o objetivo

SLA:
o compromisso contratual
```

Normalmente:

```text
SLO interno > SLA externo
```

Isso cria uma margem de segurança.

---

# 12. Error budget

Error budget representa quanto erro ou indisponibilidade é permitido pelo SLO.

Exemplo:

```text
SLO:
99,9%

Error budget:
0,1%
```

Em um mês de 30 dias:

```text
Error budget aproximado:
43,2 minutos
```

## Como usar

Se a equipe consumiu pouco error budget:

* Pode realizar mais mudanças.
* Pode testar novas funcionalidades.
* Pode aceitar mais risco controlado.

Se o orçamento foi consumido:

* Reduzir deploys arriscados.
* Priorizar confiabilidade.
* Corrigir incidentes.
* Melhorar observabilidade.

## Ideia central

Disponibilidade não significa ausência absoluta de falhas.

Significa controlar falhas dentro de um limite acordado.

# 13. RTO e RPO

## RTO

Recovery Time Objective.

Tempo máximo aceitável para restaurar o serviço.

Exemplo:

```text
RTO:
30 minutos
```

Após um desastre, o sistema precisa voltar em até 30 minutos.

## RPO

Recovery Point Objective.

Quantidade máxima aceitável de dados perdidos.

Exemplo:

```text
RPO:
5 minutos
```

O sistema pode perder no máximo cinco minutos de dados.

## Comparação

```text
RTO:
quanto tempo o sistema pode ficar fora

RPO:
quanto dado pode ser perdido
```

## Exemplo

Backups a cada 24 horas:

```text
RPO potencial:
até 24 horas
```

Replicação contínua:

```text
RPO:
segundos ou minutos
```

# 14. Escalabilidade

Escalabilidade é a capacidade de o sistema lidar com crescimento.

Esse crescimento pode ocorrer em:

* Usuários.
* Requisições.
* Dados.
* Eventos.
* Conexões.
* Regiões.
* Tenants.
* Arquivos.
* Operações por segundo.

## Escala de leitura

```text
100 mil leituras por segundo
```

## Escala de escrita

```text
20 mil escritas por segundo
```

## Escala de armazenamento

```text
2 PB em cinco anos
```

## Escala de conexões

```text
5 milhões de WebSockets simultâneos
```

Cada dimensão pode exigir soluções diferentes.

# 15. Escala vertical e horizontal

## Escala vertical

Aumentar recursos de uma máquina.

```text
8 CPUs  --> 32 CPUs
16 GB   --> 128 GB
```

### Vantagens

* Simples.
* Menos mudanças arquiteturais.
* Menor complexidade operacional.

### Desvantagens

* Limite físico.
* Máquinas maiores custam mais.
* Pode exigir downtime.
* Continua existindo um único nó.

## Escala horizontal

Adicionar máquinas.

```text
1 servidor
   |
   v
10 servidores
```

### Vantagens

* Maior capacidade.
* Melhor disponibilidade.
* Crescimento gradual.
* Menor dependência de hardware grande.

### Desvantagens

* Sistemas distribuídos.
* Balanceamento.
* Estado compartilhado.
* Concorrência.
* Consistência.
* Observabilidade mais complexa.

# 16. Throughput

Throughput é a quantidade de trabalho processada por unidade de tempo.

Exemplos:

```text
Requisições por segundo
Mensagens por segundo
Transações por segundo
Bytes por segundo
Uploads por minuto
```

## RPS

Requests per second.

```text
50.000 RPS
```

## TPS

Transactions per second.

```text
2.000 pagamentos por segundo
```

## Throughput não é latência

Um sistema pode ter:

```text
Alto throughput
+
Alta latência
```

Exemplo:

```text
Processa 1 milhão de jobs por minuto,
mas cada job leva 30 segundos.
```

Também pode ter:

```text
Baixa latência
+
Baixo throughput
```

---

# 17. Latência

Latência é o tempo necessário para concluir uma operação.

Exemplo:

```text
Cliente envia requisição:
10:00:00.000

Cliente recebe resposta:
10:00:00.180
```

Latência:

```text
180 ms
```

## Componentes da latência

```text
DNS
+
TLS
+
Rede
+
Load Balancer
+
API
+
Cache
+
Banco
+
Serviços externos
+
Serialização
```

Exemplo:

```text
Rede:             30 ms
Gateway:           5 ms
API:              10 ms
Banco:            80 ms
Serialização:      5 ms
Retorno de rede:  30 ms
-----------------------
Total:           160 ms
```

# 18. Média, mediana, P95 e P99

A média pode esconder usuários com experiência ruim.

Exemplo:

```text
99 requisições:
10 ms

1 requisição:
10.000 ms
```

A média será aproximadamente:

```text
110 ms
```

Mas um usuário esperou dez segundos.

## P50

Mediana.

```text
50% das requisições são mais rápidas.
```

## P95

```text
95% das requisições respondem até esse valor.
5% são mais lentas.
```

## P99

```text
99% respondem até esse valor.
1% é mais lento.
```

## Exemplo

```text
P50:  80 ms
P95: 250 ms
P99: 900 ms
```

## Por que percentis importam

Em grande escala, 1% pode representar muitos usuários.

```text
10 milhões de requisições
x 1%
=
100 mil requisições lentas
```

# 19. Lei de Little

A Lei de Little relaciona:

* Taxa de chegada.
* Tempo médio no sistema.
* Número médio de operações simultâneas.

Fórmula:

```text
Concorrência =
Throughput x Latência
```

Ou:

```text
L = λ x W
```

Onde:

* `L` é o número médio de itens no sistema.
* `λ` é a taxa de chegada.
* `W` é o tempo médio.

## Exemplo

```text
Throughput:
1.000 requisições por segundo

Latência média:
200 ms
=
0,2 segundo
```

```text
Concorrência =
1.000 x 0,2
=
200 requisições simultâneas
```

## Exemplo com WebSockets

Se entram:

```text
100 conexões por segundo
```

e cada conexão dura:

```text
1 hora
=
3.600 segundos
```

Número médio de conexões:

```text
100 x 3.600
=
360 mil conexões
```

# 20. Back-of-the-envelope calculations

Back-of-the-envelope calculations são estimativas rápidas feitas com aproximações.

O objetivo não é produzir um número perfeito.

O objetivo é descobrir a ordem de grandeza.

Exemplos:

```text
10 mil
100 mil
1 milhão
100 milhões
```

Essa diferença pode mudar completamente a arquitetura.

## Perguntas respondidas

* Quantas requisições por segundo?
* Quanto armazenamento será necessário?
* Quantos servidores?
* Quanto tráfego de rede?
* Quanto cache?
* Quantas partições?
* Quantas conexões?
* Quanto backlog uma fila acumulará?

# 21. Processo de estimativa

Uma sequência útil:

```text
1. Declarar hipóteses.
2. Estimar usuários.
3. Estimar ações por usuário.
4. Calcular volume diário.
5. Converter para volume por segundo.
6. Aplicar fator de pico.
7. Calcular armazenamento.
8. Calcular banda.
9. Calcular capacidade de servidores.
10. Arredondar.
```

## Exemplo

```text
Hipótese:
10 milhões de usuários ativos por dia.

Cada usuário:
20 leituras por dia.

Total:
200 milhões de leituras por dia.
```

RPS médio:

```text
200.000.000 / 86.400
≈ 2.315 RPS
```

Com pico de 10 vezes:

```text
≈ 23 mil RPS
```

# 22. Conversões úteis

## Tempo

```text
1 minuto = 60 segundos
1 hora   = 3.600 segundos
1 dia    = 86.400 segundos
```

Para estimativas rápidas:

```text
1 dia ≈ 100 mil segundos
```

Isso simplifica divisões.

## Armazenamento

Em estimativas rápidas:

```text
1 KB ≈ 1.000 bytes
1 MB ≈ 1.000 KB
1 GB ≈ 1.000 MB
1 TB ≈ 1.000 GB
1 PB ≈ 1.000 TB
```

## Rede

```text
1 byte = 8 bits
```

Portanto:

```text
100 MB/s
=
800 Mb/s
```

Adicione overhead de protocolos e margem operacional.

# 23. Estimando usuários ativos

Termos comuns:

## MAU

Monthly Active Users.

```text
Usuários ativos no mês
```

## DAU

Daily Active Users.

```text
Usuários ativos por dia
```

## Usuários simultâneos

Usuários conectados no mesmo momento.

Exemplo:

```text
100 milhões de MAU
20 milhões de DAU
2 milhões simultâneos no pico
```

Esses números representam dimensões diferentes.

## Conversão aproximada

Não assuma automaticamente:

```text
MAU = DAU
```

Pergunte ou declare uma hipótese.

Exemplo:

```text
100 milhões de MAU
20% ativos por dia
=
20 milhões de DAU
```

# 24. Estimando requisições por segundo

Considere:

```text
10 milhões de DAU
50 requisições por usuário por dia
```

Volume diário:

```text
10.000.000 x 50
=
500.000.000 requisições por dia
```

RPS médio:

```text
500.000.000 / 86.400
≈ 5.787 RPS
```

Arredondando:

```text
≈ 6 mil RPS
```

Com pico de cinco vezes:

```text
≈ 30 mil RPS
```

## Fórmula

```text
RPS médio =
Usuários diários
x ações por usuário
÷ segundos por dia
```

# 25. Média versus pico

Tráfego raramente é uniforme.

Exemplo:

```text
RPS médio:
10 mil

Pico:
80 mil
```

Fatores comuns:

* Horário comercial.
* Eventos.
* Promoções.
* Lançamentos.
* Jogos.
* Notícias.
* Black Friday.
* Virada de mês.

## Peak factor

```text
RPS de pico =
RPS médio x fator de pico
```

# 38. Capacidade versus utilização

Operar constantemente próximo de 100% é perigoso.


* Filas.
* Timeouts.
* Retries.
* Cascata de falhas.
* Aumento de latência.

## Relação entre utilização e espera

Quando um recurso se aproxima da saturação, a latência tende a crescer de forma não linear.

```text

baixa espera

espera crescente

95%:
latência alta

fila cresce continuamente
```

Muitos sistemas operam com uma meta de utilização inferior ao máximo.

Exemplo:

```text

50% a 70%
```

O valor correto depende do workload.

```text

```

Em um cluster com dez nós, o sistema deve continuar atendendo ao pico com um nó indisponível.

# 43. Trade-offs

## Disponibilidade versus consistência

Durante uma partição, pode ser necessário escolher entre:

* Continuar aceitando operações.
* Garantir consistência forte.

## Latência versus consistência

Consultar múltiplas regiões pode melhorar consistência, mas aumentar latência.

## Custo versus disponibilidade

Mais réplicas e regiões melhoram disponibilidade, mas aumentam custo.

## Capacidade versus eficiência

Muito headroom aumenta resiliência, mas pode deixar recursos ociosos.

## Precisão versus velocidade da estimativa

Estimativas rápidas ajudam na decisão inicial.

Benchmarks e testes são necessários antes de produção.

## Escala imediata versus evolução

Projetar para escala futura extrema pode gerar overengineering.

Projetar sem caminho de crescimento gera reescritas prematuras.

18. Não opere permanentemente próximo de 100%.

# 47. Questões de entrevista

## Qual a diferença entre requisito funcional e não funcional?

Requisito funcional descreve o que o sistema faz. Requisito não funcional descreve características como disponibilidade, latência, escala, consistência e segurança.

## O que significa disponibilidade de 99,9%?

Significa que o sistema pode ficar indisponível por aproximadamente 43 minutos em um mês de 30 dias.

## O que é SLI?

É uma métrica observada, como disponibilidade real ou latência P95.

## O que é SLO?

É uma meta interna para um SLI.

## O que é SLA?

É um compromisso contratual, normalmente com consequências quando não é atendido.

## O que é error budget?

É a quantidade de erro ou indisponibilidade permitida pelo SLO.

## O que é RTO?

É o tempo máximo aceitável para restaurar o serviço após uma falha.

## O que é RPO?

É a quantidade máxima aceitável de dados perdidos.

## Qual a diferença entre throughput e latência?

Throughput mede quanto trabalho é processado por unidade de tempo. Latência mede quanto tempo uma operação individual leva.

## Por que P99 importa?

Porque a média pode esconder operações muito lentas. Em grande escala, 1% pode representar milhares ou milhões de requisições.

## O que é back-of-the-envelope calculation?

É uma estimativa rápida baseada em hipóteses e arredondamentos para descobrir a ordem de grandeza do problema.

## Como calcular RPS médio?

Multiplica-se o número de usuários pelas ações por usuário e divide-se pelo número de segundos do período.

## Por que calcular o pico?

Porque o tráfego não é uniforme. Um sistema dimensionado apenas pela média pode falhar durante os horários mais movimentados.

## O que é a Lei de Little?

É a relação entre concorrência, throughput e latência:

```text
Concorrência = Throughput x Latência
```

## Por que deixar headroom?

Para absorver picos, falhas, deploys, variações de workload e tempo de reação do autoscaling.

## Uma estimativa precisa substituir um benchmark?

Não. A estimativa ajuda a escolher a direção arquitetural. Benchmarks e testes validam a capacidade real.

## Processo recomendado

```text
1. Definir escopo.
2. Identificar fluxos críticos.
3. Definir requisitos mensuráveis.
4. Declarar hipóteses.
5. Calcular volume médio.
6. Calcular pico.
7. Estimar armazenamento.
8. Estimar banda.
9. Estimar capacidade.
10. Adicionar headroom.
11. Relacionar números às decisões arquiteturais.
```

A principal ideia é:

> Estimativas não servem para prever o futuro com precisão absoluta.

Elas servem para evitar decisões arquiteturais incompatíveis com a ordem de grandeza do problema.

Uma boa solução de System Design deve ser capaz de responder:

```text
Quantos usuários?
Quantas requisições?
Qual o pico?
Quanto dado?
Qual latência?
Qual disponibilidade?
Quanto podemos perder?
Quanto tempo podemos ficar fora?
Qual é o próximo gargalo?
```
