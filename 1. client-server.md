# Client-Server

## Visão geral

A arquitetura **Client-Server**, ou **Cliente-Servidor**, é um modelo no qual dois tipos principais de componentes se comunicam por meio de uma rede:

- **Client:** solicita informações ou serviços.
    
- **Server:** recebe a solicitação, processa e devolve uma resposta.
    

```text
Client → Request → Server
Client ← Response ← Server
```

Um exemplo comum é o acesso a um site:

1. O navegador solicita uma página.
    
2. O servidor recebe a solicitação.
    
3. O servidor processa a requisição.
    
4. O servidor devolve HTML, CSS, JavaScript ou dados.
    
5. O navegador apresenta o resultado ao usuário.
    

---

## Client

O **client** é o componente utilizado pelo usuário ou por outro sistema para acessar um serviço.

Exemplos:

- Navegador web.
    
- Aplicativo mobile.
    
- Aplicativo desktop.
    
- Smart TV.
    
- Outro servidor.
    
- Dispositivo IoT.
    

Principais responsabilidades:

- Capturar ações do usuário.
    
- Enviar requisições ao servidor.
    
- Receber respostas.
    
- Exibir ou utilizar os dados recebidos.
    
- Realizar validações básicas.
    
- Manter parte do estado da interface.
    

Exemplo:

```text
Usuário clica em "Entrar"
→ Aplicação envia e-mail e senha para o servidor
→ Servidor valida as credenciais
→ Aplicação recebe o resultado
```

O client não deve ser considerado um ambiente confiável, pois o usuário pode modificar requisições, alterar o código executado no navegador ou chamar diretamente a API.

Por isso, regras importantes de segurança e negócio precisam ser validadas novamente no servidor.

---

## Server

O **server** é o componente que disponibiliza recursos, dados ou serviços para os clientes.

Principais responsabilidades:

- Receber requisições.
    
- Validar dados e permissões.
    
- Executar regras de negócio.
    
- Consultar ou modificar dados.
    
- Integrar-se com outros serviços.
    
- Retornar uma resposta ao client.
    

Exemplos de operações realizadas pelo servidor:

- Autenticar um usuário.
    
- Cadastrar um produto.
    
- Processar um pagamento.
    
- Consultar pedidos.
    
- Enviar uma mensagem.
    
- Gerar um relatório.
    

Um servidor pode atender vários clientes simultaneamente.

```text
Client A ─┐
Client B ─┼──→ Server
Client C ─┘
```

---

## Request e Response

A comunicação entre client e server normalmente acontece por meio de um modelo de **requisição e resposta**.

### Request

A requisição contém informações sobre aquilo que o client deseja realizar.

Pode incluir:

- Método da operação.
    
- Endereço do recurso.
    
- Cabeçalhos.
    
- Parâmetros.
    
- Corpo da requisição.
    
- Dados de autenticação.
    

Exemplo de requisição HTTP:

```http
GET /api/users/10
Authorization: Bearer token
```

### Response

A resposta é o resultado enviado pelo servidor.

Pode conter:

- Código de status.
    
- Cabeçalhos.
    
- Dados.
    
- Mensagem de erro.
    
- Metadados.
    

Exemplo:

```json
{
  "id": 10,
  "name": "Maria",
  "email": "maria@email.com"
}
```

---

## Protocolos

Client e server precisam seguir regras de comunicação chamadas de **protocolos**.

Protocolos comuns:

- HTTP e HTTPS.
    
- WebSocket.
    
- TCP.
    
- UDP.
    
- FTP.
    
- SMTP.
    

Em aplicações web, o protocolo mais utilizado é o **HTTPS**, que é o HTTP com criptografia.

---

## HTTP

O HTTP é um protocolo baseado em requisições e respostas.

Métodos comuns:

|Método|Utilização|
|---|---|
|`GET`|Consultar dados|
|`POST`|Criar um recurso|
|`PUT`|Atualizar um recurso completo|
|`PATCH`|Atualizar parcialmente um recurso|
|`DELETE`|Remover um recurso|

Exemplos:

```http
GET /users
POST /users
GET /users/10
PATCH /users/10
DELETE /users/10
```

---

## Status HTTP

O servidor utiliza códigos de status para informar o resultado da requisição.

### 2xx — Sucesso

- `200 OK`: operação realizada com sucesso.
    
- `201 Created`: recurso criado com sucesso.
    
- `204 No Content`: operação realizada sem conteúdo na resposta.
    

### 3xx — Redirecionamento

- `301 Moved Permanently`: recurso movido permanentemente.
    
- `302 Found`: redirecionamento temporário.
    
- `304 Not Modified`: o recurso não foi alterado.
    

### 4xx — Erro do client

- `400 Bad Request`: requisição inválida.
    
- `401 Unauthorized`: autenticação necessária ou inválida.
    
- `403 Forbidden`: usuário autenticado, mas sem permissão.
    
- `404 Not Found`: recurso não encontrado.
    
- `409 Conflict`: conflito com o estado atual do recurso.
    
- `429 Too Many Requests`: muitas requisições.
    

### 5xx — Erro do servidor

- `500 Internal Server Error`: erro interno inesperado.
    
- `502 Bad Gateway`: resposta inválida de outro servidor.
    
- `503 Service Unavailable`: serviço indisponível.
    
- `504 Gateway Timeout`: outro serviço demorou para responder.

## DNS

Normalmente, o client não conhece diretamente o endereço IP do servidor. Ele utiliza um domínio, como:

```text
api.exemplo.com
```

O **DNS**, Domain Name System, traduz esse domínio para um endereço IP.

Fluxo simplificado:

```text
Client
→ Consulta o DNS
→ Recebe o endereço IP
→ Conecta-se ao servidor
→ Envia a requisição
```

---

## IP e porta

O endereço IP identifica uma máquina na rede.

A porta identifica qual serviço dentro dessa máquina receberá a conexão.

Exemplos:

```text
192.168.0.10:80
192.168.0.10:443
192.168.0.10:8080
```

Portas comuns:

- `80`: HTTP.
    
- `443`: HTTPS.
    
- `22`: SSH.
    
- `25`: SMTP.
    
- `5432`: PostgreSQL.
    
- `3306`: MySQL.
    
- `1433`: SQL Server.

## API

Uma **API** define como o client pode se comunicar com o servidor.

Ela estabelece:

- Endpoints disponíveis.
    
- Formato das requisições.
    
- Formato das respostas.
    
- Métodos permitidos.
    
- Regras de autenticação.
    
- Possíveis erros.
    

Exemplo:

```http
GET /api/products/15
```

Resposta:

```json
{
  "id": 15,
  "name": "Notebook",
  "price": 3500
}
```

A API funciona como um contrato entre o client e o server.

## Tipos de client

### Thin Client

O client possui pouca lógica e depende fortemente do servidor.

Características:

- Processamento concentrado no servidor.
    
- Client mais simples.
    
- Atualizações centralizadas.
    
- Maior dependência da conexão com o servidor.
    

### Thick Client

O client executa uma parte maior do processamento.

Características:

- Mais lógica no dispositivo.
    
- Pode funcionar parcialmente offline.
    
- Reduz algumas chamadas ao servidor.
    
- Pode exigir atualizações no dispositivo.
    

Aplicações modernas frequentemente utilizam um modelo intermediário, dividindo responsabilidades entre client e server.

---

## Stateless e Stateful

### Stateless

Em uma arquitetura stateless, cada requisição contém todas as informações necessárias para ser processada.

O servidor não precisa lembrar das requisições anteriores.

```text
Request 1 → processada independentemente
Request 2 → processada independentemente
```

Vantagens:

- Facilita a escalabilidade.
    
- Facilita o balanceamento de carga.
    
- Reduz dependência entre requisições.
    
- Permite adicionar novos servidores com mais facilidade.
    

Exemplo:

```http
Authorization: Bearer token
```

O token é enviado em cada requisição.

### Stateful

Em uma arquitetura stateful, o servidor mantém informações sobre a sessão ou interação do cliente.

Exemplos:

- Sessão armazenada na memória do servidor.
    
- Conexão WebSocket.
    
- Jogos multiplayer.
    
- Chamadas de voz.
    

Desvantagem:

Caso o estado esteja armazenado apenas em um servidor, o usuário pode precisar continuar sendo atendido pela mesma instância.

Esse comportamento pode exigir **sticky sessions** ou um armazenamento de sessão compartilhado.

---

## Escalabilidade

Com o aumento do número de clientes, um único servidor pode não suportar toda a carga.

### Escalabilidade vertical

Consiste em aumentar os recursos da máquina existente.

Exemplos:

- Mais memória.
    
- Mais CPU.
    
- Disco mais rápido.
    

```text
Servidor pequeno → Servidor maior
```

Vantagens:

- Mais simples.
    
- Poucas mudanças na arquitetura.
    

Desvantagens:

- Existe um limite físico.
    
- Pode ser caro.
    
- Mantém um único ponto de falha.
    

### Escalabilidade horizontal

Consiste em adicionar mais servidores.

```text
Server 1
Server 2
Server 3
```

Vantagens:

- Maior capacidade de crescimento.
    
- Melhor tolerância a falhas.
    
- Permite distribuir a carga.
    

Desvantagens:

- Aumenta a complexidade.
    
- Exige distribuição de tráfego.
    
- Pode exigir sincronização de dados e estados.
    

---

## Load Balancer

O **Load Balancer**, ou balanceador de carga, distribui as requisições entre vários servidores.

```text
Clients
   ↓
Load Balancer
   ↓
Server 1
Server 2
Server 3
```

Responsabilidades:

- Distribuir o tráfego.
    
- Evitar sobrecarga em uma única instância.
    
- Remover servidores indisponíveis da distribuição.
    
- Melhorar disponibilidade.
    
- Permitir escalabilidade horizontal.
    

Estratégias comuns:

- Round Robin.
    
- Least Connections.
    
- Weighted Round Robin.
    
- Hash de IP.
    

---

## Banco de dados

O servidor geralmente acessa um banco de dados para persistir informações.

```text
Client → Server → Database
Client ← Server ← Database
```

O client normalmente não deve acessar diretamente o banco de dados.

O servidor funciona como uma camada de proteção e controle, sendo responsável por:

- Validar permissões.
    
- Aplicar regras de negócio.
    
- Evitar exposição de credenciais.
    
- Controlar consultas e alterações.
    
- Registrar operações.
    

---

## Cache

O cache armazena dados temporariamente para reduzir o tempo de resposta e evitar processamento repetido.

O cache pode existir em diferentes locais:

- Navegador.
    
- CDN.
    
- Load Balancer.
    
- Aplicação.
    
- Banco de dados.
    
- Sistema distribuído, como Redis.
    

Exemplo:

```text
Client solicita produto
→ Servidor consulta o cache
→ Se existir, retorna o dado
→ Se não existir, consulta o banco
→ Salva o resultado no cache
→ Retorna ao client
```

---

## Comunicação síncrona

Na comunicação síncrona, o client envia uma requisição e espera pela resposta.

```text
Client → Request → Server
Client ← Response ← Server
```

Exemplos:

- HTTP.
    
- REST.
    
- GraphQL.
    
- gRPC síncrono.
    

Vantagens:

- Simples de implementar.
    
- Resposta imediata.
    
- Fluxo fácil de compreender.
    

Desvantagens:

- O client fica esperando.
    
- Uma lentidão no servidor afeta o client.
    
- Pode criar forte dependência entre serviços.
    

---

## Comunicação assíncrona

Na comunicação assíncrona, o client ou serviço envia uma solicitação sem precisar aguardar todo o processamento.

Exemplo:

```text
Client → Server → Queue
                    ↓
                  Worker
```

O servidor pode responder imediatamente:

```json
{
  "status": "processing"
}
```

Exemplos de uso:

- Envio de e-mails.
    
- Processamento de vídeos.
    
- Geração de relatórios.
    
- Processamento de pagamentos.
    
- Notificações.
    
- Tarefas demoradas.
    

---

## Polling

No polling, o client consulta o servidor periodicamente para verificar se existe alguma atualização.

```text
Client → Existe atualização?
Server → Não

Client → Existe atualização?
Server → Não

Client → Existe atualização?
Server → Sim
```

Vantagem:

- Simples de implementar.
    

Desvantagens:

- Pode gerar requisições desnecessárias.
    
- A atualização não é instantânea.
    
- Pode aumentar a carga no servidor.
    

---

## Long Polling

No long polling, o servidor mantém a requisição aberta até existir uma atualização ou ocorrer um timeout.

```text
Client → Solicita atualização
Server → Mantém conexão aberta
Server → Envia resposta quando houver dado
```

Após receber a resposta, o client cria uma nova requisição.

É mais eficiente que o polling tradicional, mas ainda possui o custo de abrir novas requisições.

---

## WebSocket

O WebSocket cria uma conexão persistente e bidirecional entre client e server.

```text
Client ⇄ Server
```

Tanto o client quanto o servidor podem enviar mensagens a qualquer momento.

Casos de uso:

- Chats.
    
- Jogos online.
    
- Notificações em tempo real.
    
- Acompanhamento de entregas.
    
- Cotações financeiras.
    
- Edição colaborativa.
    

Em comparação ao HTTP tradicional, o WebSocket é mais adequado quando existem atualizações frequentes e em tempo real.

---

## Latência

A latência representa o tempo necessário para os dados viajarem entre client e server.

Ela pode ser afetada por:

- Distância geográfica.
    
- Qualidade da conexão.
    
- Quantidade de intermediários.
    
- Tempo de processamento.
    
- Consultas ao banco de dados.
    
- Chamadas para outros serviços.
    
- Tamanho da resposta.
    

Para reduzir a latência, podem ser utilizados:

- Cache.
    
- CDN.
    
- Compressão.
    
- Servidores próximos aos usuários.
    
- Otimização de consultas.
    
- Redução do tamanho das respostas.
    
- Processamento assíncrono.
    

---

## Disponibilidade

Disponibilidade indica quanto tempo o sistema permanece acessível.

Um servidor único representa um **Single Point of Failure**, pois, caso ele pare, todo o serviço fica indisponível.

Para aumentar a disponibilidade, podem ser utilizados:

- Múltiplos servidores.
    
- Load Balancer.
    
- Replicação de banco de dados.
    
- Monitoramento.
    
- Health checks.
    
- Failover.
    
- Distribuição em diferentes zonas ou regiões.
    

---

## Segurança

Na arquitetura client-server, alguns cuidados são essenciais:

- Utilizar HTTPS.
    
- Autenticar usuários.
    
- Validar permissões.
    
- Não confiar nos dados enviados pelo client.
    
- Validar entradas no servidor.
    
- Evitar exposição de dados sensíveis.
    
- Aplicar rate limiting.
    
- Proteger contra SQL Injection.
    
- Proteger contra XSS e CSRF.
    
- Armazenar senhas utilizando hash seguro.
    
- Utilizar tokens com expiração.
    

A interface pode esconder um botão, mas isso não representa segurança.

Mesmo que o client não apresente uma determinada funcionalidade, o servidor precisa verificar se o usuário possui permissão para executar a operação.

## Exemplo completo

Considere um aplicativo de compras.

### Fluxo para consultar um produto

```text
1. Usuário abre o aplicativo.
2. Client consulta o DNS.
3. DNS retorna o IP do servidor.
4. Client estabelece uma conexão HTTPS.
5. Client envia GET /products/10.
6. Load Balancer escolhe uma instância.
7. Servidor verifica o cache.
8. Caso necessário, consulta o banco de dados.
9. Servidor monta a resposta.
10. Client recebe os dados.
11. Aplicativo exibe o produto.
```

### Fluxo para realizar uma compra

```text
1. Client envia os itens do carrinho.
2. Servidor autentica o usuário.
3. Servidor valida preços e estoque.
4. Servidor processa o pagamento.
5. Servidor registra o pedido.
6. Servidor publica tarefas assíncronas.
7. E-mail de confirmação é enviado.
8. Servidor retorna o pedido criado.
```

O servidor deve recalcular preços e validar o estoque. Ele não deve confiar nos valores enviados pelo client.

---

## Vantagens da arquitetura Client-Server

- Centralização das regras de negócio.
    
- Melhor controle de segurança.
    
- Compartilhamento de recursos.
    
- Dados centralizados.
    
- Facilidade de manutenção do back-end.
    
- Possibilidade de atender diferentes tipos de client.
    
- Permite escalabilidade horizontal.
    
- Facilita integrações entre sistemas.
    

---

## Desvantagens

- Dependência da rede.
    
- O servidor pode se tornar um gargalo.
    
- Falhas no servidor podem afetar vários clientes.
    
- Escalabilidade pode aumentar a complexidade.
    
- Problemas de latência afetam a experiência do usuário.
    
- Exige cuidados com segurança, concorrência e disponibilidade.
    

---

## Pontos importantes para System Design

Ao projetar uma arquitetura client-server, algumas perguntas devem ser consideradas:

- Quantos clientes utilizarão o sistema?
    
- Qual será o volume de requisições?
    
- O sistema precisa funcionar em tempo real?
    
- As operações serão síncronas ou assíncronas?
    
- O servidor será stateless ou stateful?
    
- Como o sistema será escalado?
    
- Existe necessidade de Load Balancer?
    
- Onde o estado será armazenado?
    
- Como serão tratados erros e timeouts?
    
- O sistema precisa de cache?
    
- Qual nível de disponibilidade é necessário?
    
- Como autenticação e autorização serão realizadas?
    
- Como evitar que uma falha derrube todo o sistema?

```text

Server:
- Recebe as requisições.
- Valida dados e permissões.
- Executa regras de negócio.
- Acessa bancos e outros serviços.
- Devolve respostas.

Elementos importantes:
- DNS.
- IP e portas.
- HTTP e HTTPS.
- API.
- Request e response.
- Banco de dados.
- Cache.
- Load Balancer.
- Escalabilidade.
- Segurança.
- Disponibilidade.
```

A arquitetura client-server é a base de grande parte dos sistemas modernos. Em System Design, ela é o ponto inicial para compreender como usuários, aplicações, servidores, bancos de dados e serviços distribuídos se comunicam.
